毕业设计开发文档：未知网络流量应用协议识别系统
1. 系统概述
本系统针对真实校园网环境下的网络流量审计需求，解决“未知应用协议识别”问题。核心目标是通过预处理过滤已知流量，对剩余未知流量使用智能体（Agent）进行主动多工具分析，最终输出复合标签（如 bilibili:video、unknown_news:web），并可视化各协议/应用的流量占比。

问题场景：明文流量直接深挖具体服务类型；加密流量优先识别SNI（已知SNI直接判断类型）；SNI未知/未标记时，触发主动获取更多数据（如网页内容、证书细节、IP查询、流量行为统计）。
输入：真实校园网原始PCAP文件（批量离线处理）。
输出：复合标签统计 + 可视化图表（饼图/柱状图/Sankey图，展示应用层 vs 服务类型层比例）。
创新点：结合传统DPI工具过滤 + LLM驱动的Agent主动推理，实现“已知快速过滤 + 未知智能挖掘”的两阶段架构，支持双维度（应用层 + 服务类型层）标签。

2. 系统目标与功能需求

功能目标：
准确过滤已知流量，减少Agent处理量。
对未知流量实现复合标签识别：应用层（bilibili、QQ音乐、微信、unknown_app等） + 服务类型层（video、audio、file、chat、news、control等）。
支持冲突解决：多源证据（如SNI、模型输出、网页内容）融合，优先SNI/主动获取结果。
输出审计报告：流量比例可视化 + 每个未知流量的推理日志与置信度。

性能目标（本科可控）：离线处理校园网PCAP，支持中等规模数据（GB级）；准确率以可视化结果合理性为主要评价。
非功能：使用开源工具为主，可复现；集成LangSmith进行Agent调试与评估。

3. 整体系统架构（文字流程图）
text[输入层]
├── 真实校园网原始 PCAP 文件

[预处理层 - 已知流量过滤（模块A）]
├── 流聚合 + 元数据提取（SNI、证书、端口、统计特征）
├── 开源DPI/日志工具过滤（nDPI + Zeek 为主）
│   ├── 明文流量 → 直接识别具体服务类型 → known:xxx:service_type
│   ├── 加密 + SNI已知/可标记 → 直接映射标签（known:bilibili:video）
│   └── SNI未知/未匹配/置信低 → 未知流量集合（带元数据）
├── 输出：已知标签集合 + 未知流量集合

[核心处理层 - Agent智能挖掘（模块B，使用改造的mybot）]
├── 输入：未知流量文件 + 元数据
├── Agent：
│   ├── 场景判断（SNI已知 / SNI未知）
│   ├── Tool/MCP调用（并行或依照一个合理的工作流）：
│   │   - 复现分类模型（辅助，低权重，统计特征输入）
│   │   - SNI/证书/IP查询/WHOIS工具
│   │   - 网页访问/内容分析工具（主动获取，SNI未知时重点触发）
│   ├── 融合与冲突解决节点（优先SNI/主动获取结果，LLM推理 + 置信加权）
│   ├── 双维度标签生成节点：
│   │   - 应用层：优先SNI/证书/域名
│   │   - 服务类型层：优先内容/行为特征（允许灵活顺序）
│   └── 输出：复合标签 + 解释日志 + 置信度
└── 合并已知 + 未知结果

[输出层]
├── 标签统计与聚合
├── 可视化模块（Matplotlib/Plotly：饼图、柱状图、应用-服务Sankey图）
└── 日志报告（推理过程）

4. 模块详细设计
模块A：已知流量过滤（预处理）

目标：快速处理90%以上常见流量，减负Agent。
流程：PCAP → 流聚合 → 元数据提取 → 工具过滤 → 已知/未知分割。
结合导师建议：明文直接深挖；加密SNI已知直接判；未知留给主动Agent。
实现要点：
维护校园常见SNI/域名/证书白名单（从历史流量或公开列表起步）。
设置置信阈值：高置信→已知；低/冲突→未知。

输出格式：结构化日志/JSON（已知带标签，未知带原始元数据）+ 筛选出来的未知流量pcap文件（做为分类模型的输入）。

模块B：Agent核心（未知流量处理）

框架：mybot agent框架
核心逻辑（符合导师“明确场景 + 主动获取”）：
明确不同场景的处理方式(skills解决)：SNI已知但未命中白名单的加密流量；SNI未知的加密流量。
工具集（先不用实现具体工具，留接口即可）：
分类模型工具：模型（ISCX上强，真实PCAP作软参考，低权重）。
网络查询工具：SNI解析、证书分析、IP归属、WHOIS。
主动获取工具：网页访问（标题/关键词提取）、流量统计。
外部MCP导入

冲突解决：SNI/主动成功时优先其结果；失败时提升统计+证书权重；LLM输出解释链与置信分。
双维度推断：不强制严格顺序，谁证据强先用谁，再补全另一维。

Prompt设计原则：明确场景、工具使用规则、优先级（SNI > 主动内容 > 模型）、输出结构化标签。

模块C：后处理与可视化

统计复合标签比例。
可视化：应用层占比、 服务类型层占比、跨维度关系图。
日志：每个未知流推理trace（便于论文实验分析）。


5. 潜在挑战与应对

挑战1：真实PCAP上模型效果差 → 应对：降级为低权重辅助，优先SNI/主动工具。
挑战2：多源冲突（SNI vs 模型） → 应对：场景化优先级 + LLM解释 + 置信加权。
挑战3：SNI未知/不可访问 → 应对：主动工具补救 + 流量统计特征。
挑战4：工程量与本科周期 → 应对：先实现核心（Zeek过滤 + 简单Agent），再迭代工具。
挑战5：数据隐私/规模 → 应对：离线处理，匿名化测试；从小批量PCAP起步。

6. 实验与评估计划

数据集：真实校园网PCAP（主） + ISCX等开源（复现验证 + 对比）。
评估方式：已知部分用工具准确率；未知部分用人工抽样验证标签合理性 + 可视化结果分析；记录推理过程评估Agent决策质量。
消融实验：有/无Agent、有/无主动工具、不同融合策略对比。
论文支撑：流程图、工具集成细节、冲突案例分析、校园流量审计可视化结果。

每个文件/目录的具体功能（详细说明）

README.md
项目整体介绍、实验目标（未知流量审计 + 复合标签 + 可视化比例）。
安装步骤（Zeek、nDPI、Python依赖）。
运行示例命令。
如何添加新SNI到白名单、常见问题排查。

requirements.txt
核心依赖

config.py
定义路径：ZEek_BIN_PATH, NDPI_READER_PATH, KNOWN_SNI_PATH, DATA_DIR 等。
LLM配置（本地模型或API）。
置信度阈值、白名单加载逻辑。

data/
存放输入PCAP、白名单、输出结果。方便实验重复。

scripts/（命令行入口，快速实验）
run_preprocessing.py：输入一个或多个PCAP路径 → 调用Zeek生成日志 → nDPI补充特征 → 过滤已知/未知 → 保存结果JSON。

core/preprocessing.py（模块A核心）
主函数：process_pcap(pcap_path)
内部流程：
调用Zeek（zeek -C -r pcap_path）生成日志。
解析 conn.log（流聚合）、ssl.log（SNI、证书）、http.log（明文）。
调用nDPI（ndpiReader -i pcap_path -C output.csv）补充协议、统计特征、SNI。
加载known_sni_list.json进行匹配。
按导师建议分场景：明文直接识别、SNI已知直接打标签、其余标记为未知（带完整元数据）。

输出：已知结果列表 + 未知流量列表（每个包含uid、SNI、证书、统计特征等）。

core/agent/（模块B核心）


core/utils/
zeek_utils.py：执行Zeek命令、用pandas或zeek-cut解析日志、提取SNI等字段。
ndpi_utils.py：执行ndpiReader，解析CSV/JSON输出，提取协议、SNI、字节分布等。
helpers.py：通用辅助（标签映射、置信度计算、日志记录）。

core/visualization.py
使用Plotly生成三种图表：应用层饼图、服务类型层柱状图、Sankey关系图。
支持保存为文件或返回JSON（便于Notebook展示）。

notebooks/
强烈推荐在这里做大多数实验：一步步调试Zeek输出、测试冲突案例、比较有/无Agent的结果、记录人工验证准确率等。
便于生成论文中的图表和实验截图。
